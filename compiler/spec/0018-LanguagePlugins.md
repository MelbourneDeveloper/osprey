# Language Plugins

Language plugins let an Osprey program embed expressions written in a foreign language (such as SQL, GraphQL, or a regex dialect) inside an ordinary function declaration. The plugin parses, type-checks, and lowers the embedded text at compile time. The result is a normal Osprey function with a static type, a static signature, and compile-time errors for malformed embedded code.

A plugin is an external executable that the compiler invokes during code generation. The compiler and the plugin communicate over stdin/stdout using JSON.

## Plugin Function Declaration

A plugin function is a top-level declaration that names the plugin, the function, the parameters, an optional return-type specification, and the body in the foreign language.

```ebnf
pluginFnDecl     ::= docComment? "fn" pluginName functionName
                     "(" paramList? ")" pluginReturnType? "=" pluginContent ";"
pluginName       ::= IDENT
functionName     ::= IDENT
pluginReturnType ::= "->" type    (* existing Osprey type *)
                   | "as"  type   (* type generated by the plugin *)
pluginContent    ::= (any token except ";")+
```

The terminating `;` distinguishes a plugin declaration from an ordinary `fn`. Parameter syntax is identical to ordinary functions.

### Examples

```osprey
// No declared return type; plugin infers it
fn sql getUsers(userType: int) =
    SELECT id, name, email FROM users;

// Returns an existing Osprey type
fn sql getUserById(userId: int) -> User =
    SELECT * FROM users WHERE id = $userId;

// Plugin generates a new type named UserProfile
fn sql getUserProfile(userId: int) as UserProfile =
    SELECT users.name, users.email, profiles.bio
    FROM users JOIN profiles ON users.id = profiles.user_id
    WHERE users.id = $userId;
```

## Return-Type Modes

| Form         | Meaning                                                                 |
| ------------ | ----------------------------------------------------------------------- |
| (omitted)    | The plugin reports the return type. The compiler uses the reported type. |
| `-> T`       | The function returns the existing type `T`. The plugin must agree.       |
| `as T`       | The plugin generates a new type named `T` and uses it as the return type.|

In all cases the effective return type is wrapped in `Result<T, E>`; see [Plugin Errors](#plugin-errors).

## Plugin Discovery

The compiler looks for plugins under `<compilerDir>/plugins/<pluginName>/`. The compiler directory is resolved in this order:

1. The `OSPREY_COMPILER_DIR` environment variable, if set.
2. The parent directory of the running `osprey` binary (the binary lives at `<compilerDir>/bin/osprey`).
3. The current working directory.

Within a plugin directory the compiler looks for, in order:

1. A pre-built executable named `osprey-<pluginName>-plugin`.
2. A Go module (a `go.mod` file). If found, the plugin is invoked as `go run .`.

A plugin directory must contain a `config.json` file (see [Plugin Configuration](#plugin-configuration)).

## Compile-Time Protocol

For each plugin function declaration the compiler:

1. Constructs a `PluginRequest` and serialises it as JSON.
2. Spawns the plugin executable.
3. Writes the request to the plugin's stdin and closes stdin.
4. Reads a single JSON `PluginResponse` from the plugin's stdout.
5. Waits for the plugin to exit. A non-zero exit code, malformed JSON, or `success: false` is a compilation error.

### Request

```json
{
  "functionName": "getUsers",
  "languageBody": "SELECT id, name FROM users WHERE status = $status",
  "parameters":   [{"name": "status", "type": "string"}],
  "context":      {"database_url": "postgresql://localhost:5432/postgres"},
  "filePath":     "src/users.osp",
  "lineNumber":   42
}
```

| Field          | Description                                                              |
| -------------- | ------------------------------------------------------------------------ |
| `functionName` | The function name from the declaration.                                  |
| `languageBody` | The verbatim text between `=` and `;`.                                   |
| `parameters`   | Declared parameters in source order: `{name, type}`.                     |
| `context`      | Plugin-specific values supplied by the compiler (see [Context](#context)). |
| `filePath`     | Path of the source file containing the declaration.                      |
| `lineNumber`   | 1-based line of the declaration; used in error messages.                 |

### Response

```json
{
  "success": true,
  "returnType": {
    "type": "Result",
    "genericParams": [
      {"type": "Array", "genericParams": [
        {"type": "Record", "fields": [
          {"name": "id",    "type": "int"},
          {"name": "name",  "type": "string"}
        ]}
      ]},
      {"type": "DatabaseError"}
    ]
  },
  "generatedCode": {
    "imports":    ["std.database", "std.sql"],
    "fiberCode":  "<Osprey source emitted as the function body>",
    "errorTypes": ["ConnectionError", "QueryError", "TimeoutError"]
  }
}
```

On failure the plugin returns:

```json
{ "success": false, "error": "missing parameter $status referenced by query" }
```

The compiler reports the `error` string at the source location of the declaration.

### Context

The compiler populates `context` based on `pluginName`. Built-in mappings:

| Plugin                     | Variable                | Default                                  |
| -------------------------- | ----------------------- | ---------------------------------------- |
| `postgresql`, `postgres`   | `OSPREY_POSTGRESQL_URL` | `postgresql://localhost:5432/postgres`   |
| `mysql`                    | `OSPREY_MYSQL_URL`      | `mysql://localhost:3306/mysql`           |
| `sqlite`                   | `OSPREY_SQLITE_URL`     | `sqlite://./database.db`                 |

Other plugins receive an empty `context`.

## Plugin Errors

Plugin functions implicitly return `Result<T, E>` where `E` is the union of error types listed in `generatedCode.errorTypes`. Callers must pattern-match the result; see [Error Handling](0013-ErrorHandling.md).

```osprey
fn sql getUser(id: int) -> User =
    SELECT * FROM users WHERE id = $id;

let result = getUser(id: 1)
match result {
    Success { value } => print("Found: ${value.name}")
    Error   { message } => print("Lookup failed: ${message}")
}
```

## Plugin Configuration

Each plugin directory contains a `config.json`:

```json
{
    "plugin_name": "sql",
    "version": "1.0.0",
    "description": "SQL integration plugin for Osprey",
    "executable": "osprey-sql-plugin",
    "supported_languages": ["sql", "postgresql", "mysql", "sqlite"],
    "configuration": {
        "connection_pool_size": 10,
        "query_timeout_ms": 30000
    },
    "error_types": ["ConnectionError", "QueryError", "TimeoutError"]
}
```

Required fields: `plugin_name`, `version`, `executable`. All other fields are advisory and consumed by the plugin itself.

## Security

Plugins run as ordinary subprocesses with the compiler's privileges. The compiler does not sandbox plugin execution. A project that invokes a plugin trusts that plugin to the same degree it trusts its build tooling. Sandbox flags described in [Security and Sandboxing](0016-SecurityAndSandboxing.md) restrict the generated program's runtime, not the plugin's compile-time execution.

## Implementation Status

Implemented:

- Grammar and AST for plugin function declarations.
- Plugin discovery, subprocess invocation, and JSON request/response.
- Reference SQL plugin under `compiler/plugins/sql/` covering `SELECT`, `INSERT`, `UPDATE`, `DELETE`, parameter validation, and column-type inference.

Not yet wired through code generation:

- The LLVM backend currently emits a placeholder for plugin call sites instead of invoking the generated body. The plugin is consulted (and its validation enforced), but the value returned at runtime is not yet the value the plugin would produce.

A program that uses plugin declarations will compile when the plugin validates the body, but runtime behaviour of plugin calls is incomplete until backend integration lands.
